# Registries: Discovery for skilltree

**Version**: 1.0 (spec)
**Date**: 2026-03-30

Registries add discovery and search to skilltree. They are **authoring-time tools** — they help you find and add skills. They are never in the install path, never in the resolution path, and never required for `skilltree install` or `skilltree update` to work.

## The Problem

Today, adding a remote skill requires knowing the exact git repo URL and path within it:

```bash
skilltree add python-coding --repo github.com/org/shared-skills --path skills/python-coding --version "^2.0.0"
```

There is no way to search for a skill by name, browse what's available in a repository, or discover skills across multiple repositories. You must already know where a skill lives before you can use it.

## Design Principle

**Registries are lookup tools, not runtime dependencies.**

A registry helps you at `skilltree add` time. After that, the manifest is self-contained. This follows `go get` — it resolves module paths at add time, but `go.mod` records the full path. It does NOT follow npm's model where the manifest says `"lodash": "^4.0.0"` and a global registry is needed to resolve it.

This means:
- `skilltree.yaml` always contains explicit `repo:` + `path:` for remote deps
- `skilltree install` never consults registries
- `skilltree update` never consults registries
- A teammate without registries configured can still install, update, and develop
- The manifest remains auditable — you can read it and know exactly where every dependency comes from

## How It Works

### Registries are git repos

A registry is just a git repo that contains skills and/or agents. Any existing skill repo is already a valid registry — no changes needed. skilltree clones the repo and scans it to build a searchable index.

### Global config: `~/.skilltree/config.yaml`

```yaml
registries:
  - name: shared-skills
    repo: github.com/org/shared-skills
  - name: community
    repo: github.com/skilltree/community-skills
  - name: internal
    repo: github.com/mycompany/private-skills
```

This file is user-level. It is never checked into a project. Different developers can have different registries configured — this is fine because registries only affect discovery, not resolution.

### Optional index: `skillkit-index.yaml`

A repo can include a `skillkit-index.yaml` at its root to make search fast:

```yaml
# skillkit-index.yaml -- optional, speeds up search
entities:
  - name: python-coding
    type: skill
    path: skills/python-coding
    description: "Python development with Poetry, PEP 8, pytest, type hints"
    tags: [python, testing, backend]
  - name: code-review
    type: skill
    path: skills/code-review
    description: "Automated code review with linting and testing"
    tags: [review, testing, quality]
  - name: deploy
    type: skill
    path: skills/deploy
    description: "Build and deploy automation workflows"
    tags: [deploy, ci, automation]
```

**Fields:**

| Field | Required | Description |
|-------|----------|-------------|
| `name` | Yes | Entity name (from frontmatter or directory name) |
| `type` | Yes | `skill` or `agent` |
| `path` | Yes | Path within the repo |
| `description` | No | One-line summary (from frontmatter `description:` field) |
| `tags` | No | Searchable keywords |

If a repo does not have this file, skilltree falls back to **dynamic scanning** — walking `git ls-tree -r HEAD` for `SKILL.md` files and `.md` agent files, reading their frontmatter. This is slower but means any skill repo works as a registry with zero changes.

**Who generates the index?** The repo maintainer, ideally via CI (e.g., a GitHub Action that runs `skilltree index` on push and commits the result). Hand-maintaining it is discouraged — it drifts. See the `skilltree index` command below.

### Cache layout

```
~/.skilltree/
  config.yaml                  # Global registries
  cache/                       # Existing: bare clones for dependency resolution
    github.com/
      imarios/
        shared-skills/
  registry-cache/              # New: for search (separate from dep cache)
    shared-skills/
      repo/                    # Bare git clone
      index.json               # Parsed search index (generated by registry update)
    community/
      repo/
      index.json
    internal/
      repo/
      index.json
```

Registry cache is separate from dependency cache because they have different lifecycles:
- **Dependency cache**: fetched on-demand during `skilltree install`, governed by lockfile
- **Registry cache**: fetched on TTL (default 24h) or explicit `skilltree registry update`

The `index.json` file is the parsed, queryable search index. It is generated by `skilltree registry update` and read by `skilltree search`. This separates the expensive work (git fetch + scan) from the fast work (query matching).

## Commands

### `skilltree registry add <url> [--name <alias>]`

Register a git repo as a searchable registry.

```bash
$ skilltree registry add github.com/org/shared-skills
  Added registry 'shared-skills' (github.com/org/shared-skills)

$ skilltree registry add github.com/mycompany/private-skills --name internal
  Added registry 'internal' (github.com/mycompany/private-skills)
```

**Name inference:** If `--name` is omitted, uses the last path segment of the URL (e.g., `shared-skills` from `github.com/org/shared-skills`). If the name conflicts with an existing registry, error with suggestion to use `--name`.

**Writes to:** `~/.skilltree/config.yaml`

**Does not:** clone the repo immediately. Cloning happens on first `search` or `registry update`.

### `skilltree registry remove <name>`

```bash
$ skilltree registry remove internal
  Removed registry 'internal'
  Cleaned registry cache
```

Removes from config and deletes the registry cache for that name.

### `skilltree registry list`

```bash
$ skilltree registry list
  Name        Repo                                   Entities  Last Updated
  shared-skills       github.com/org/shared-skills               23        2 hours ago
  community   github.com/skilltree/community-skills    47        1 day ago
  internal    github.com/mycompany/private-skills      8        never (run 'skilltree registry update')
```

`Entities` and `Last Updated` show `--` if the registry hasn't been fetched yet.

### `skilltree registry update [name]`

Fetch (or re-fetch) registry repos and rebuild the local search index.

```bash
$ skilltree registry update
  Updating shared-skills... 23 entities (12 skills, 11 agents)
  Updating community... 47 entities (31 skills, 16 agents)
  Updating internal... 8 entities (6 skills, 2 agents)

$ skilltree registry update shared-skills
  Updating shared-skills... 23 entities (12 skills, 11 agents)
```

**Steps:**
1. `git fetch` (or `git clone --bare` if first time) into `~/.skilltree/registry-cache/<name>/repo/`
2. Check for `skillkit-index.yaml` at HEAD — `git show HEAD:skillkit-index.yaml`
3. **If found** — parse it. Each entry provides `name`, `type`, `path`, `description`, `tags`.
4. **If not found** — dynamic scan:
   - `git ls-tree -r HEAD` to list all paths in the repo
   - Filter for `SKILL.md` files (parent directory = skill path) and standalone `.md` files (potential agents)
   - For each match, `git show HEAD:<path>` to read frontmatter
   - Extract `name` (from frontmatter or directory name), `type` (skill/agent), `path`, `description`
   - Dynamic scan won't produce `tags` since those aren't in the SKILL.md standard
5. Write the parsed index to `~/.skilltree/registry-cache/<name>/index.json` with a timestamp

The `index.json` format:

```json
{
  "registry": "shared-skills",
  "repo": "github.com/org/shared-skills",
  "updated_at": "2026-03-31T10:30:00Z",
  "entities": [
    {
      "name": "python-coding",
      "type": "skill",
      "path": "skills/python-coding",
      "description": "Python development with Poetry, PEP 8, pytest, type hints",
      "tags": ["python", "testing", "backend"]
    }
  ]
}
```

`search` reads `index.json` directly — it never touches the git repo.

### `skilltree search <query>`

Search across all registered registries.

```bash
$ skilltree search python
  python-coding          skill   shared-skills       Python development with Poetry, PEP 8, pytest
  python-testing         skill   community   Python test utilities and fixtures
  → skilltree add <name> --repo <repo> --path <path>

$ skilltree search security --registry internal
  internal-scanner       skill   internal    Custom security scanner integration
  alert-triage           agent   internal    SOC alert triage workflow
  → skilltree add <name> --repo <repo> --path <path>
```

**Flags:**
- `--registry <name>` — search only one registry
- `--type skill|agent` — filter by entity type
- `--json` — machine-readable output

**Output includes the full `skilltree add` command** so the user can copy-paste:

```bash
$ skilltree search "code review"
  code-review      skill   shared   Automated code review with linting and testing
  → skilltree add code-review --repo github.com/org/shared-skills --path skills/code-review

  review-standards skill   shared   Team code review standards and conventions
  → skilltree add review-standards --repo github.com/org/shared-skills --path skills/review-standards
```

**Stale cache:** If a registry hasn't been fetched (or was last fetched more than 24h ago), `search` prints a warning but still uses the cached data if available. It does NOT auto-fetch — that would make `search` unpredictably slow.

```
  Warning: registry 'community' last updated 3 days ago. Run 'skilltree registry update community' for latest.
```

**No cache at all:** If a registry has never been fetched (`index.json` doesn't exist), it is skipped with a message:

```
  Skipping registry 'internal' (never updated). Run 'skilltree registry update internal' first.
```

#### Search algorithm

The data pipeline: `registry update` writes `index.json` → `search` reads `index.json` and queries in-memory. The git repo is never touched during search.

**Tokenization:** The query is split into lowercase tokens. `"python testing"` → `["python", "testing"]`. Single-word queries produce one token.

**Matching:** All tokens must match somewhere in the entity (AND semantics). If any token has zero hits across the entity's name, description, and tags, the entity is excluded. This prevents `"python kubernetes"` from returning `python-coding` just because "python" matched the name.

**Scoring:** Each token is scored against each field, and scores are summed across all tokens:

| Match | Score | Example |
|-------|-------|---------|
| Exact name match (full query = name) | 100 | query `python-coding`, name `python-coding` |
| Name contains token | 10 | query `python`, name `python-coding` |
| Tag exact match | 5 | query `testing`, tags `[..., testing, ...]` |
| Description contains token | 1 | query `python`, description "Python development with..." |

All matching is case-insensitive.

**Sorting:** Results ordered by total score descending. Ties broken alphabetically by name.

**Complexity:** Linear scan over all entities across all registries. No inverted index, no fuzzy matching, no stemming. The dataset is small — a large registry has 100-200 entities, and a developer might have 3-5 registries configured. A linear scan with string matching over ~500 entities is sub-millisecond.

### `skilltree info <name>`

Show detailed information about a specific entity across registries.

```bash
$ skilltree info python-coding
  python-coding (skill)
  Registry:     shared-skills (github.com/org/shared-skills)
  Path:         skills/python-coding
  Description:  Python development with Poetry, PEP 8, pytest, type hints
  Tags:         python, testing, backend
  Dependencies: general-coding
  Versions:     3.1.0, 3.0.2, 3.0.1, 3.0.0, 2.1.3, 2.0.0
  Latest:       3.1.0

  → skilltree add python-coding --repo github.com/org/shared-skills --path skills/python-coding --version "^3.1.0"
```

If the name exists in multiple registries, show all matches:

```bash
$ skilltree info python-coding
  Found in 2 registries:

  [1] python-coding (skill) — shared-skills
      github.com/org/shared-skills :: skills/python-coding
      Python development with Poetry, PEP 8, pytest, type hints

  [2] python-coding (skill) — community
      github.com/skilltree/community-skills :: skills/python-coding
      Community Python coding skill

  → skilltree add python-coding --repo <repo> --path <path>
```

### `skilltree add <name>` (enhanced — registry-assisted)

When `skilltree add` is called with just a name (no `--repo`, `--source`, or `--local`), it consults registries to resolve the full coordinates:

```bash
# One match — writes full form to manifest
$ skilltree add python-coding
  Resolved from registry 'shared-skills': github.com/org/shared-skills/skills/python-coding
  Added python-coding to dependencies (version: "*")

# What gets written to skilltree.yaml:
# python-coding:
#   repo: github.com/org/shared-skills
#   path: skills/python-coding
#   version: "*"
```

```bash
# Multiple matches — interactive disambiguation
$ skilltree add python-coding
  'python-coding' found in 2 registries:
    [1] shared-skills      — github.com/org/shared-skills :: skills/python-coding
    [2] community  — github.com/skilltree/community-skills :: skills/python-coding
  Select [1-2] or use --repo to specify: 1
  Added python-coding to dependencies (version: "*")
```

```bash
# No match — helpful error
$ skilltree add nonexistent-skill
  Error: 'nonexistent-skill' not found in any registry
  Run 'skilltree search <query>' to find available skills
  Or specify --repo and --path explicitly
```

**Important:** The enhanced `add` always writes the **full explicit form** to `skilltree.yaml`. The registry is consulted once, at add time, and then forgotten. The manifest never depends on registry state.

**Flag interaction:**
- `--repo` or `--source` or `--local` → existing behavior, registries not consulted
- No location flag → registry lookup
- `--version` → applied after registry resolution (default: `"*"`)
- `--registry <name>` → only search that registry (skips disambiguation)

### `skilltree index` (authoring command)

Generate `skillkit-index.yaml` for a skill repo. Intended for repo maintainers.

```bash
$ cd my-skills-repo
$ skilltree index
  Scanned 23 entities (12 skills, 11 agents)
  Wrote skillkit-index.yaml

$ skilltree index --check
  skillkit-index.yaml is up to date ✓

$ skilltree index --check
  skillkit-index.yaml is stale (2 entities changed, 1 added)
  Run 'skilltree index' to update
  Exit 1
```

**CI integration:**
```yaml
# .github/workflows/index.yml
- run: skilltree index --check
  # Fails if index is stale — author must regenerate
```

## What This Does NOT Change

- **`skilltree.yaml` format** — no new fields, no shorthand syntax. Remote deps still require `repo:` + `path:`.
- **`skilltree.lock` format** — unchanged.
- **`skilltree install`** — never consults registries. Lockfile-first behavior unchanged.
- **`skilltree update`** — resolves from manifest `repo:` URLs, not registries.
- **`sources:` section** — still URL aliases, not searchable, not registries.
- **Resolution algorithm** — manifest → context → same-repo → error. No registry step.

## Relationship to `sources:`

`sources:` and registries are separate concepts that serve different purposes:

| | `sources:` | Registries |
|---|---|---|
| **Scope** | Project (`skilltree.yaml`) | User (`~/.skilltree/config.yaml`) |
| **Purpose** | URL alias (avoid repeating `repo:` URLs) | Discovery (find skills you don't know about) |
| **Used by** | `skilltree install`, `skilltree update` | `skilltree search`, `skilltree add` (without `--repo`) |
| **Required for install?** | Yes (expanded during resolution) | No (never consulted) |
| **Checked into git?** | Yes | No |

A source alias and a registry can point to the same repo — that's fine. They're independent features that happen to reference the same git URL.

## Implementation Phases

### Phase A: Infrastructure
- `~/.skilltree/config.yaml` parsing/writing
- `skilltree registry add`, `remove`, `list`
- Registry cache management (clone, fetch, separate from dep cache)

### Phase B: Search
- `skilltree registry update` (fetch + scan/parse index)
- `skillkit-index.yaml` parsing
- Dynamic scanning fallback (git ls-tree + frontmatter reading)
- `skilltree search` with matching and formatted output
- `skilltree info`
- `skilltree index` (authoring command)

### Phase C: Enhanced Add
- `skilltree add <name>` without `--repo` — registry-assisted resolution
- Interactive disambiguation for multiple matches
- `--registry` flag for explicit registry selection
- Error messages with `search` suggestions

## Design Decisions

### 1. Registries are discovery-only
Registries never appear in the install or resolution path. The manifest is always self-contained. Rationale: avoids implicit external state, prevents "works on my machine" from registry ordering, keeps the manifest auditable.

### 2. `skilltree add` writes full form
Even when registry-assisted, `add` writes `repo:` + `path:` to the manifest. The registry is consulted once and then irrelevant. Rationale: the manifest must be the complete source of truth for resolution.

### 3. Registry cache is separate from dependency cache
Different lifecycles (TTL vs on-demand), different purposes (search vs resolution), separate directories. Avoids cross-contamination.

### 4. Dynamic scanning as fallback
Any git repo with skills works as a registry without adding `skillkit-index.yaml`. The index is a performance optimization, not a requirement. Rationale: zero barrier to entry. A team can `skilltree registry add` their existing skill repo and start searching immediately.

### 5. `skillkit-index.yaml` is a CI artifact
Generating the index is an authoring concern, checked with `skilltree index --check` in CI. Not hand-maintained. Rationale: hand-maintained indexes drift from reality, producing wrong search results (worse than slow results).

### 6. No auto-fetch on search
`skilltree search` uses cached data even if stale, with a warning. Rationale: search should be fast and predictable. `registry update` is the explicit fetch. Users shouldn't be surprised by network calls during search.

### 8. Parsed index cached to disk
`registry update` writes a parsed `index.json` alongside the bare repo. `search` reads this file — it never touches git. This separates the expensive work (git fetch, tree walk, frontmatter parsing) from the cheap work (string matching over a JSON array). Both `skillkit-index.yaml` (repo-provided) and dynamic scanning (fallback) produce the same `index.json` output — search doesn't know or care which path was used.

**Rejected alternative:** re-parse on every search. Simple (no extra state) but makes search slow for repos without `skillkit-index.yaml` — reading 50+ frontmatters via `git show` on every query is noticeable. Caching makes search consistently fast regardless of whether the repo provides an index.

### 7. Interactive disambiguation, not ordering priority
When the same skill name exists in multiple registries, `skilltree add` prompts the user to choose instead of silently picking by declaration order. Rationale: implicit ordering priority creates hard-to-debug behavior differences between developers.
